## 微前端

### 基础部分

1. 什么是微前端？微前端的核心思想是什么？

定义：将大型单体前端应用拆分为独立开发、部署、运行的子应用，通过主框架聚合的技术体系。
核心思想：微前端的核心思想是将前端应用拆分为多个独立的子应用，每个子应用可以独立开发、测试和部署。通过主应用进行聚合和路由管理。

2. 微前端的优缺点是什么？

优点：
- 独立开发：每个子应用可以独立开发、测试和部署，减少了团队之间的依赖。
- 技术栈独立：每个子应用可以使用不同的技术栈，适应不同的业务需求。
- 部署独立：每个子应用可以独立部署，减少了整体应用的部署风险。
- 版本独立：每个子应用可以独立升级，减少了整体应用的版本风险。
- 渐进式升级：可以逐步将单体应用拆分为微前端架构，降低了重构的风险。
- 复用能力：可以复用已有的子应用，减少了重复开发的工作量。

缺点：
- 复杂性增加：微前端架构增加了系统的复杂性，需要更多的协调和管理。
- 性能问题：多个子应用的加载和渲染可能会影响性能，需要优化。
- 共享状态管理：多个子应用之间的状态管理和数据共享可能会变得复杂。
- 版本冲突：多个子应用可能会使用不同版本的依赖库，导致版本冲突。

3. 微前端的架构模式有哪些？各自的优缺点是什么？

- `IFrame`：通过 `iframe` 加载子应用，隔离性好，但性能差，`SEO` 不友好。
- `Web Components`：使用 Web Components 技术实现子应用的封装和加载，隔离性好，但浏览器兼容性差。
- `JavaScript`：通过 JavaScript 动态加载子应用，性能好，但隔离性差，容易出现全局变量冲突。
- `Server-Side Rendering`：在服务器端渲染子应用，性能好，但实现复杂。

### `Qiankun`

- 深入理解沙箱机制（快照沙箱和代理沙箱，解释它们如何隔离子应用的全局变量，避免冲突）
- 应用加载与渲染流程（主应用如何注册、加载子应用，子应用生命周期管理）。

### `Module Federation`

- 深入理解模块联邦的原理和机制，包括共享模块、远程模块的加载和执行。
- 主应用和子应用之间的通信机制，包括共享状态、事件总线等。
- 如何在主应用中配置和加载子应用，以及如何在子应用中配置和加载主应用。
- 如何解决模块联邦中的依赖冲突和版本问题。
