# 对多语言新增国家支持的落地和思考

## 一、背景与目标

在多语言、多区域产品中，“新增一个国家”通常不只是加一条语言配置，而是一次跨模块联动：

- 路由与语言
- 请求分流与网关策略
- 地图/设计能力
- 业务规则与展示差异
- 回归与发布治理

这份文档用于沉淀可复用的方法论，帮助团队在后续扩国家时做到：

- 交付更快
- 风险更低
- 回归可控
- 知识可传承

## 二、核心结论（先看这 6 条）

1. 新增国家的第一步不是改页面，而是先做“国家能力建模”。
2. 所有国家码别名必须走统一归一化，不允许在页面或组件中重复维护。
3. 先改“配置中心”，再改“接入层”，最后改“业务层”。
4. 能力差异应配置化（开关/参数），不要用散落的 if/else。
5. 回归按“链路”执行，而不是按页面执行。
6. 每次新增国家都要产出一页复盘，持续降低后续接入成本。

## 三、推荐实施流程（可直接复用）

### 阶段 0：前置决策

先冻结以下信息：

- 国家主码（ISO 2 位）
- 路由前缀
- 支持语言组合
- 国家码别名策略（例如展示码与接口码不一致）
- 服务分区（如 AU 集群 / EU 集群）
- 业务差异范围（支付、税、设计参数、功能开关）

产出：

- 一页决策记录（负责人、决策时间、生效范围）

### 阶段 1：配置中心先行

建立或更新统一国家配置中心，建议最少包含：

- `normalizeCountryCode`
- `clusterByCountry`（请求分区）
- `routePrefixByCountry`
- `designProfileByCountry`
- `featureFlagsByCountry`
- `paymentProfileByCountry`

原则：

- 新增国家时，优先改配置中心。
- 未经评审，不允许直接在业务组件写国家分支。

### 阶段 2：基础接入层

依次处理：

1. i18n 与路由注册
2. 请求头与网关分流
3. 地图/设计引擎国家参数映射
4. 输入组件国家归一化（例如手机号组件）

验收重点：

- 国家码在请求链路中一致
- 路由前缀与语言展示一致

### 阶段 3：业务差异层

按能力域逐个接入：

- 项目创建能力开关
- 支付字段与校验
- Proposal / PDF 展示差异
- 其他国家专属业务规则

建议做法：

- 用 profile 映射替代散落分支
- 对“默认兜底”逻辑加显式说明，避免掩盖遗漏

### 阶段 4：回归与发布

按链路回归：

1. 登录与路由链路
2. 请求与分流链路
3. 设计工具链路
4. 支付与文档链路
5. 历史国家兼容链路

发布建议：

- 先灰度一个国家或一个租户
- 监控关键指标 24-72 小时后再全量

## 四、为什么常见做法会失败

### 1) 把“国家支持”当成“文案翻译”

表现：只改语言包，忽略请求分流与业务规则。

后果：线上出现“看起来可用，但关键流程失败”。

### 2) 国家码映射散落在多个模块

表现：A 模块用 `UK`，B 模块用 `GB`。

后果：跨模块状态不一致，排障成本高。

### 3) 业务分支先于配置设计

表现：页面上快速加 if/else。

后果：短期可交付，长期难维护，新增国家越来越慢。

## 五、可复用设计模式

### 模式 A：国家能力 Profile

按国家输出统一 profile，例如：

- `cluster`: `AU | EU`
- `paymentProfile`: `LOCAL | IBAN_SWIFT | CUSTOM`
- `features`: `showInvoiceNumber`, `hideAutoDetect` 等
- `designParams`: 设计引擎参数与阈值

### 模式 B：分层映射

- 第一层：国家码归一化
- 第二层：国家 -> profile
- 第三层：profile -> UI/校验/参数

优势：

- 业务层只关心 profile，不关心具体国家码。

### 模式 C：默认策略 + 显式例外

- 默认走通用逻辑
- 少量国家作为显式例外配置
- 例外必须在文档中登记

## 六、对外可共享的回归清单

### 1) 基础一致性

- 路由前缀正确
- 语言加载正确
- 国家码归一化正确

### 2) 请求链路

- 请求头国家码正确
- 分流到正确服务集群
- 关键 API 全部通过

### 3) 业务链路

- 项目创建可用
- 设计流程可用
- 支付信息可用
- 对外文档（Proposal/PDF）可用

### 4) 兼容性

- 历史国家不回归
- 别名链路（如 UK/GB）一致

## 七、组织与协作建议

- 设立“国家接入负责人”，统一跨团队接口。
- 评审时强制检查“是否先改配置中心”。
- 每次新增国家完成后，更新一次能力矩阵。

建议维护三类资产：

1. 国家能力矩阵（配置视角）
2. 回归记录模板（质量视角）
3. 复盘记录（组织视角）

## 八、复盘模板（可复制）

- 本次新增国家：
- 生效范围：
- 改动模块：
- 风险点：
- 实际事故/告警：
- 根因：
- 已修复项：
- 可复用资产：
- 下一次可减少的工作量：

## 九、实践建议（下一步）

如果团队准备持续扩国家，建议按优先级推进：

1. 统一国家能力配置中心（若未完全统一）。
2. 将高频国家分支迁移为 profile 映射。
3. 把回归清单接入发布流程（门禁化）。
4. 每次接入后更新能力矩阵与复盘文档。

## 十、位置级落地清单（半脱敏版）

以下给出“到模块位置”的落地点，便于快速实施，同时避免暴露过多内部细节。

### A. 配置层（必须先改）

1. 统一国家配置模块
	- 位置示例：`constants/countryProfile`
	- 典型改动：国家码归一化、集群分流、路由前缀、功能开关、支付 profile、设计参数映射。
2. i18n 区域与语言注册
	- 位置示例：`config/modules/extensions`、`composables/regionLocale`
	- 典型改动：新增 locale、region 语言映射、默认语言兜底。

### B. 接入层（高风险）

1. 请求插件与网关分流
	- 位置示例：`plugins/httpRequest`
	- 典型改动：请求头国家码、AU/EU 分流、别名统一处理。
2. 地图与设计入口
	- 位置示例：`composables/useMap`、`pages/.../design`
	- 典型改动：地图服务分流、设计引擎国家参数映射。
3. 输入组件归一化
	- 位置示例：`components/PhoneInput`
	- 典型改动：初始化国家码复用统一归一化函数。

### C. 业务层（按能力域推进）

1. 登录/注册路由落点
	- 位置示例：`layers/accounts/pages/sign-in`、`layers/accounts/pages/sign-up`、`layers/accounts/components/SignupForm`
2. 项目创建差异开关
	- 位置示例：`pages/.../CreateProjectDrawer`
3. 支付与文档输出
	- 位置示例：`pages/settings/payment`、`proposal`、`pdf` 支付详情组件
4. 其他国家特性
	- 位置示例：税率、发票号、融资、设计阈值等相关设置页与组件

### D. 验收层（上线前）

1. 路由与语言链路
2. 请求与分流链路
3. 设计与地图链路
4. 支付与文档链路
5. 历史国家兼容链路

## 十一、一次新增国家的最小改动包（建议模板）

如果新增一个国家，推荐最小改动包按下面交付：

1. 配置中心改动
	- 国家码归一化映射
	- 集群分流映射
	- 路由前缀映射
	- 能力开关映射
2. i18n 改动
	- locale 注册
	- 语言映射与文案补齐
3. 接入层改动
	- 请求头与网关分流
	- 地图/设计参数映射
4. 业务层改动
	- 注册/登录跳转
	- 项目创建差异
	- 支付字段与校验
	- Proposal/PDF 展示
5. 质量产物
	- 回归记录
	- 风险说明
	- 上线观察项

## 十二、可复用检查问题（评审时直接问）

1. 国家码别名是否只在统一配置模块维护？
2. 是否存在新增页面 if/else 但未配置化的分支？
3. 请求层与地图层分流是否同时更新？
4. 支付与文档输出是否同步更新？
5. 旧国家是否做了最小回归验证？
6. 是否补充了本次国家接入复盘？
